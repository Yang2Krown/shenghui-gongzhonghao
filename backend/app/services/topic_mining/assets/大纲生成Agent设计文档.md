# 大纲生成 Agent 设计文档 v1.0

> **用途**：定义大纲生成模块的架构、Agent 分工、输入输出格式、传播标注规范与自检机制。
> **与其他文档的关系**：上游接 [选题套路库](./选题套路库.md) + [选题评分卡](./选题评分卡.md)；下游会被「正文生成 Agent」和「标题生成 Agent」消费。
> **维护者**：路人甲TM 团队
> **最后更新**：2026-05-20
> **版本说明**：v1.0 为 4 Agent 协作版本，中粒度大纲输出，固定权重自检。

---

## 0. 设计原则

### 0.1 大纲生成解决什么问题
从一条已入选的选题，产出一份**符合公众号传播规律**的中粒度大纲（小标题 + 信息点 + 字数预估 + 传播角色标注 + 自检评分）。

### 0.2 三条不可妥协的原则

1. **大纲是注意力曲线而非逻辑展开**：不是按"是什么 → 怎么做 → 总结"的论文思路，而是按"读者注意力高低起伏"来排兵布阵。
2. **每一节必须有明确的传播角色**：开头钩子 / 价值密度 / 转发理由 / 收藏点 / 结尾升华，缺一项自检会扣分。
3. **多 Agent 协作 > 单 Agent 一锅炖**：单个 Agent 容易给"稳妥但平庸"的答案，多 Agent 各司其职能拉高质量上限。

### 0.3 文档结构
- 第 1 章：整体架构与数据流
- 第 2-5 章：4 个 Agent 的详细职责与提示词框架
- 第 6 章：大纲的标准输出格式（中粒度）
- 第 7 章：传播触发点标注规范
- 第 8 章：完整工作流与失败处理
- 第 9 章：配套资产
- 第 10 章：维护与迭代

## 1. 整体架构

### 1.1 4 Agent 协作图

```
                输入：一条已入选的选题
                  （含标题、套路、方向、价值承诺、原始信息）
                              │
                              ▼
        ┌────────────────────────────────────────┐
        │  Step 0  方向识别 → 调用模板库（如有）   │
        └────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────┐
│  Agent A  大纲创作员（百万粉 AI 博主视角）                  │
│  ✓ 必须产出 3 个候选大纲                                   │
│  ✓ 3 个候选必须使用不同的开头钩子类型（强制差异化）          │
└──────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────┐
│  Agent B  大纲评审员（公众号运营专家视角）                  │
│  ✓ 从 3 个候选中选最优 / 必要时融合两个候选的优点           │
│  ✓ 为每一节补全传播角色标签                                 │
└──────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────┐
│  Agent C  读者挑刺员（目标读者代入）                        │
│  ✓ 以核心读者身份模拟阅读                                   │
│  ✓ 标出"会跳读 / 会划走 / 觉得无聊"的节，给出修改建议        │
└──────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────┐
│  Agent D  大纲自检员（按 6 特征打分）                       │
│  ≥ 7 分 → ✅ 通过，输出最终大纲                            │
│  < 7 分 → 把扣分理由喂给 Agent A 重生（最多 2 次）           │
│  2 次仍不过 → ❌ 丢回人工，标记"难以成稿"                   │
└──────────────────────────────────────────────────────────┘
                              │
                              ▼
                        输出最终大纲
```

### 1.2 数据流

```
选题对象（Selection）
  → 3 个候选大纲（OutlineCandidates）
  → 1 个评审后大纲（ReviewedOutline，含传播标签）
  → 1 个挑刺后大纲（RevisedOutline，已根据读者视角修改）
  → 1 个自检后大纲（FinalOutline，含评分）
```

### 1.3 性能预算（参考）

| Agent | 模型 | 单次预估 token | 备注 |
|-------|------|--------------|------|
| Agent A | Claude/GPT 主模型 | ~3000 (in) + ~3000 (out) | 输出 3 候选，token 较多 |
| Agent B | 同上 | ~3500 (in) + ~1200 (out) | 输入是 3 候选 |
| Agent C | 同上 | ~2000 (in) + ~800 (out) | 单候选挑刺 |
| Agent D | 同上 | ~2000 (in) + ~600 (out) | 评分输出 |
| **总计** | | ~10500 (in) + ~5600 (out) | 单条选题正常流程 |

如果触发重生，token 大约翻倍。建议每条选题大纲成本预算控制在 0.3 USD 以内（按 Claude Sonnet 价格）。

## 2. Agent A - 大纲创作员

### 2.1 角色定位
**百万粉 AI 公众号博主**，深谙公众号读者注意力规律，擅长把一条选题拆成多种叙事骨架。

### 2.2 输入
```yaml
选题:
  标题: ...
  方向: 实践型 / 解决问题型 / 教程型 / 观点型 / 整活型 / 资讯型
  套路: 来自套路库（如 1.1.6 避坑/性价比型）
  价值承诺: ...
  原始信息: ...
模板参考:
  - （来自大纲模板库，如果该方向有对应模板）
```

### 2.3 核心工作规则

**规则 1 - 必须产出 3 个候选**
不是 1 个，不是 5 个。3 是经过权衡的——足够多以保证多样性，足够少以控制成本和评审复杂度。

**规则 2 - 3 个候选必须强制差异化**
- 每个候选必须使用**不同的开头钩子类型**（从下面 6 种里选 3 种）：
  - 痛点共鸣 / 反差冲击 / 场景代入 / 数据冲击 / 悬念开头 / 反共识开头
- 3 个候选的**整体骨架**（节数、节奏、价值阶梯）也应明显不同
- 不允许出现 "A、B、C 都是开头痛点共鸣，只是换了说法"

**规则 3 - 中粒度输出**
每节包含：小标题 + 核心信息点（bullet） + 字数预估 + 备注（暂不要求传播标签——这是 Agent B 的工作）

**规则 4 - 字数与节数约束**
- 总字数 **2000-3000**，**目标值 ≥ 2500**（这是高质量公众号正文的基本盘，字数低于 2500 难以承载足够的"价值密度 + 案例 + 金句"）
- 节数 4-6
- 单节 400-600 字最舒服

**规则 5 - 不允许学究式小标题**
- ❌ "一、引言""二、Claude Goal 功能介绍"
- ✅ "三个月烧了 2000 块订阅费后，我才发现 Claude Goal 是个分水岭"

### 2.4 提示词框架

```
你是一位百万粉的 AI 公众号博主，擅长从选题切入不同叙事骨架。

【输入选题】
{selection}

【模板参考】
{template_if_any}

【你的任务】
为这条选题生成 3 个差异化的中粒度大纲候选。

【硬约束】
1. 3 个候选必须使用 3 种不同的开头钩子类型，从以下 6 种选：
   痛点共鸣 / 反差冲击 / 场景代入 / 数据冲击 / 悬念开头 / 反共识开头
2. 3 个候选的骨架结构必须明显不同（不允许只是改字、不动骨架）
3. 每个候选总字数 2000-3000，**目标 ≥ 2500**，节数 4-6
4. 单节字数 400-600
5. 小标题禁止学究式（不用"一、二、三、引言、结论"）
6. 输出严格按下面的 YAML 格式

【输出格式】
（参考第 6 章定义的标准格式）

【自我检查清单】
□ 3 个候选用的开头钩子是否真的不同？
□ 3 个候选的骨架是否明显不同？
□ 每节字数和总字数是否在约束范围内？
□ 小标题是否避免了学究式？
```

### 2.5 重生模式（第二次调用时）

第一次输出未通过自检时，Agent A 会被**重新调用**，输入额外加上 Agent D 的扣分理由：

```
【第二次调用，前次失败原因】
{deduction_reasons}

【请针对性改进】
请在新的 3 个候选中重点解决上述问题，但仍保持 3 个候选的差异化。
```

## 3. Agent B - 大纲评审员

### 3.1 角色定位
**公众号运营专家**，看过 1000+ 篇爆款和扑街，能判断"哪个骨架最有传播潜力"。

### 3.2 输入
- Agent A 输出的 3 个候选大纲
- 原始选题对象

### 3.3 核心工作

**任务 1 - 选最优 / 融合**
- 多数情况下选 1 个最优候选
- 但允许"融合"——如果 A 候选的开头最强、B 候选的中段最有料、C 候选的结尾升华最好，可以做剪接组合

**任务 2 - 补全传播角色标签**
为最终大纲的每一节标 1-2 个传播角色（标签清单见第 7 章）。**强制要求**：
- 至少包含一节"开头钩子"
- 至少包含一节"价值密度"
- 至少包含一节"转发理由"
- 至少包含一节"收藏点"

**任务 3 - 微调小标题与节奏**
评审过程中可以微调小标题文字，让标签和小标题更匹配。

### 3.4 提示词框架

```
你是一位运营过百万粉公众号的资深运营专家。

【输入：3 个候选大纲】
{candidates}

【输入：原始选题】
{selection}

【你的任务】
从 3 个候选中选出最有传播潜力的版本，必要时可融合多个候选的优点形成新版本，
并为最终大纲的每一节标注传播角色。

【评估视角】
- 开头能不能在 3 行内钩住读者？
- 每节是否都有"新增价值"？
- 是否有强烈的情绪节点（焦虑/反差/共鸣）？
- 是否有可收藏可转发的具体内容？

【传播角色标签清单】（每节标 1-2 个，必须从清单里选）
开头钩子 / 痛点共鸣 / 价值密度 / 案例展示 / 反差或反共识 /
金句 / 操作步骤 / 工具清单 / 转发理由 / 结尾升华

【强制要求】
最终大纲必须包含至少一节"开头钩子"、一节"价值密度"、
一节"转发理由"、一节"收藏点"（操作步骤或工具清单）。

【输出格式】
（参考第 6 章定义的最终大纲格式）

【评审说明】
请同时输出"为什么选这个 / 为什么这么融合"的 1-2 句话理由。
```

## 4. Agent C - 读者挑刺员

### 4.1 角色定位
**目标读者本人**——25-35 岁、互联网产品/运营/研发、对 AI 有一定基础。代入这个身份去"挑刺"。

### 4.2 输入
- Agent B 输出的评审后大纲

### 4.3 核心工作

**任务 - 以读者身份模拟阅读，找出"会跳/会划走"的节**

挑刺维度：
1. **看到这个开头，我会继续看吗？** 不会 → 第 1 节问题大
2. **看完第 N 节，我有动力看第 N+1 节吗？** 没有 → 第 N 节有问题
3. **哪节给我的感觉是"我已经知道了"？** → 这节没新增价值
4. **哪节让我感觉"作者在水"？** → 这节信息密度低
5. **看到结尾，我会想转发或收藏吗？** 不会 → 结尾问题大

### 4.4 输出格式

```yaml
读者挑刺反馈:
  整体感受: ...（1-2 句话总评）
  问题节列表:
    - 节号: 2
      问题类型: 信息密度低 / 没有新增 / 读起来累 / 跳脱主线 / ...
      具体反馈: ...
      修改建议: ...
  修订后大纲: (含针对性调整后的完整大纲)
```

### 4.5 提示词框架

```
你现在的身份是一位 28 岁的互联网产品经理，男性，工作 5 年，
日常使用 Claude / ChatGPT / Cursor 等 AI 工具，关注 AI 行业动态。
你订阅了"路人甲TM"公众号，今天打开了一篇文章。

【这篇文章的大纲】
{reviewed_outline}

【你的任务】
以这位读者的真实视角，逐节阅读这个大纲，挑出会让你"想划走""觉得在水"
"已经知道了"的节，并给出具体的修改建议。然后输出一个修订后的大纲版本。

【挑刺时请诚实】
不要客套。如果这个大纲对你来说就是平庸的，请直说"我读到第 3 节就关了"。
不要为了"政治正确"给好评——我们要的就是你的真实反应。

【输出格式】
（参考第 4.4 节）
```

## 5. Agent D - 大纲自检员

### 5.1 角色定位
独立评分员，按"好大纲 6 特征"给出量化评分，不参与创作。

### 5.2 输入
- Agent C 输出的修订后大纲

### 5.3 评分维度与锚点

#### 维度 1 - 开头钩子强度（权重 20%）

| 分数 | 标准 |
|------|------|
| 9-10 | 第 1 节命中 6 种钩子类型之一，且力度强烈，3 行内必有冲击 |
| 7-8 | 命中钩子类型，但力度温和 |
| 4-6 | 钩子模糊，需要读者忍耐才能进入 |
| 1-3 | 无明显钩子，从背景或定义开头 |

#### 维度 2 - 价值阶梯递进（权重 20%）

| 分数 | 标准 |
|------|------|
| 9-10 | 每节都有清晰的"新增价值"，且类型多样（信息/方法/视角/案例交替） |
| 7-8 | 大部分节有新增，少数节略弱 |
| 4-6 | 有 1-2 节明显原地踏步 |
| 1-3 | 多节重复，读者读完只记住开头 |

#### 维度 3 - 节奏感（权重 15%）

| 分数 | 标准 |
|------|------|
| 9-10 | 明显的紧/松、抽象/具体、信息/故事交替 |
| 7-8 | 有节奏但不够明显 |
| 4-6 | 节奏较平，但能撑下去 |
| 1-3 | 全程匀速密集，读到一半就累 |

#### 维度 4 - 小标题扫读友好度（权重 15%）

| 分数 | 标准 |
|------|------|
| 9-10 | 每个小标题脱离正文都能独立传递信息，且有钩子感 |
| 7-8 | 小标题清晰但偏陈述式 |
| 4-6 | 部分小标题模糊或学究化 |
| 1-3 | 学究式小标题（"引言""结论"等） |

#### 维度 5 - 传播触发点完整度（权重 20%）

| 分数 | 标准 |
|------|------|
| 9-10 | 必备 4 类标签齐全（开头钩子+价值密度+转发理由+收藏点），且分布合理 |
| 7-8 | 4 类齐全但分布略集中 |
| 4-6 | 缺 1 类 |
| 1-3 | 缺 2 类及以上 |

#### 维度 6 - 长度与节数匹配（权重 10%）

| 分数 | 标准 |
|------|------|
| 10 | 总字数 **2500-3000**，节数 4-6，单节 400-600 |
| 8 | 总字数 2000-2499，节数 4-6，单节字数合规 |
| 5 | 总字数 1700-1999 或单节字数明显偏离（< 300 或 > 700） |
| 1 | 总字数 < 1700 或 > 3500，或节数 < 3 或 > 8 |

### 5.4 总分公式

```
总分 = 0.20×钩子 + 0.20×价值阶梯 + 0.15×节奏感
     + 0.15×小标题 + 0.20×传播触发点 + 0.10×长度
```

### 5.5 通过门槛与重生
- **总分 ≥ 7.0**：✅ 通过，输出最终大纲
- **总分 < 7.0**：触发重生流程
  - 第 1 次失败 → 将"扣分理由"作为额外输入，重新调用 Agent A
  - 第 2 次失败 → 不再尝试，标记"难以成稿"，丢回人工

### 5.6 输出格式

```yaml
自检评分:
  开头钩子强度: 8
  价值阶梯递进: 9
  节奏感: 8
  小标题扫读友好度: 9
  传播触发点完整度: 9
  长度与节数匹配: 10
  总分: 8.7
判定: ✅ 通过
扣分理由: （仅在不通过时输出）
  - 维度: 开头钩子强度
    扣了几分: 4
    原因: 第 1 节开头是 Claude Goal 功能介绍，缺乏冲击点
    建议: 改用反差或痛点共鸣开头
```

## 7. 传播触发点标注规范

### 7.1 10 个固定标签（不可自创）

| 标签 | 触发的读者行为 | 适合放在 |
|------|-------------|--------|
| **开头钩子** | 完读率 | 第 1 节 |
| **痛点共鸣** | 完读率 + 转发 | 第 1-2 节 |
| **价值密度** | 完读率 + 收藏 | 中段节 |
| **案例展示** | 信任感 | 中段节 |
| **反差或反共识** | 完读率 + 转发 | 中后段节 |
| **金句** | 转发 | 末段节 |
| **操作步骤** | 收藏 | 中后段节 |
| **工具清单** | 收藏 | 中后段节 |
| **转发理由** | 转发 | 末段节 |
| **结尾升华** | 转发 + 评论 | 末节 |

### 7.2 4 类必备标签（缺一扣 5 分）

| 必备类别 | 涵盖标签 |
|---------|---------|
| 开头钩子 | 开头钩子 / 痛点共鸣（任一） |
| 价值密度 | 价值密度 / 操作步骤 / 工具清单（任一） |

### 7.3 标签规则
- 每节 1-2 个标签
- 同一标签可在多节出现
- 不允许自创标签（保证下游可统计）
- 新增标签需要走人工审议、追加到本文档

## 8. 完整工作流与失败处理

### 8.1 主流程时序

```
T+0    输入：1 条入选选题
T+1    Step 0 - 方向识别 + 调用模板（若有）
T+2    Agent A 第 1 次调用 → 3 个候选大纲
T+3    Agent B 评审 → 1 个评审后大纲（含传播标签）
T+4    Agent C 挑刺 → 1 个修订后大纲
T+5    Agent D 自检 → 评分

       ├─ 总分 ≥ 7  → ✅ 输出最终大纲（结束）
       └─ 总分 < 7  → 进入重生流程

T+6    Agent A 第 2 次调用（含 Agent D 扣分理由）→ 3 新候选
T+7    Agent B + Agent C + Agent D 再跑一遍

       ├─ 总分 ≥ 7  → ✅ 输出最终大纲（结束）
       └─ 总分 < 7  → ❌ 标记"难以成稿"，丢回人工
```

### 8.2 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| Agent A 没产出 3 个候选 | 重试 1 次；仍不行则降级单候选并标记异常 |
| Agent A 的 3 个候选未做差异化 | Agent B 主动指出，要求 Agent A 重生 |
| Agent C 反馈"全篇都不想看" | 视为自检 0 分，触发重生 |
| 连续 2 次自检失败 | 丢回人工，记录失败原因供后续分析 |
| API 调用失败 | 重试最多 3 次，仍失败标记选题状态为"待重试" |

### 8.3 输出归档

每条选题的大纲生成产物都需归档（用于后续训练/复盘）：
- 3 个候选大纲（原始）
- 评审后大纲
- 读者挑刺反馈
- 修订后大纲
- 自检评分明细
- 是否重生、重生次数
- 是否最终通过

- 标题生成 Agent 设计文档（待建）— 与大纲并行产出多标题候选

### 9.3 大纲模板库的使用方式
当大纲模板库建成后，Agent A 的工作流变成：
1. 读取选题的"方向"字段
2. 查询模板库 → 拿到该方向的 2-3 个经典模板
3. 把模板作为"参考骨架"放进 Agent A 的提示词
4. Agent A 仍然产出 3 个差异化候选，但其中至少 1 个应基于模板改造

模板库的接入是"参考"而不是"强制"——Agent A 始终保有创造空间。

## 10. 维护与迭代

### 10.1 Agent 提示词的优化
- 实战中发现某个 Agent 经常给出某类错误（如 Agent A 总是开头痛点共鸣），在提示词里加针对性约束
- 每月回顾一次各 Agent 的输出质量，识别系统性偏差

### 10.2 评分锚点的优化
- 实战中如果某个评分维度的锚点不准（同样输出，人工和 Agent D 评分严重不一致），调整锚点描述
- 锚点的调整记录都要写进版本日志

### 10.3 权重的优化
- v1.0 的权重（20/20/15/15/20/10）是基于讨论拍的
- 运行 1-2 个月后，根据真实发布数据反推权重——比如发现"开头钩子强的文章爆款率显著更高"，可以提高该维度权重

### 10.4 一票否决项的新增
- 实战中如果发现某类大纲必败（如某些方向用了某些骨架），追加到自检的"硬规则"中

### 10.5 反馈回流
每篇文章发布后，人工标注：
- 大纲哪几节实际表现好（被读者评论提及、被引用）
- 大纲哪几节表现差（评论区抱怨、阅读流失点）
- 大纲的自检分数与最终爆款度的相关性

这些数据每月做一次回归分析，反向校准 Agent。
