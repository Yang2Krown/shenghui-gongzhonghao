> **用途**：定义标题生成模块的架构、Agent 分工、输入输出格式、套路分布约束与自检机制。
> **与其他文档的关系**：上游接 [选题套路库](./选题套路库.md) / [大纲生成 Agent 设计文档](./大纲生成Agent设计文档.md) / [标题套路库](./标题套路库.md)；下游会被「正文生成 Agent」消费（保证正文与标题承诺一致）。
> **维护者**：路人甲TM 团队
> **最后更新**：2026-05-20

## 0. 设计原则

### 0.1 标题生成解决什么问题
公众号的打开率几乎完全由标题决定。**没人点开，写得再好也是 0**。这个 Agent 的任务是：基于已通过自检的选题和大纲，产出 3 个推荐标题供人工最终选定。

### 0.2 五条不可妥协的原则

1. **打开率优先于正确性**：标题必须**先抓人**再讲信息。一篇关于"AI 替代客服"的文章，标题"AI 客服系统应用"和"我的客服朋友最近开始焦虑了"，前者准确但没人看。
2. **多候选 + 多套路 > 单候选反复打磨**：一个套路反复打磨 5 个版本，不如 10 个套路各出 1 个版本。多样性比精修更重要。
3. **标题必须兑现承诺**：标题承诺什么，正文必须给到什么。"3 个用法"标题对应正文必须有清晰可见的 3 个用法。
4. **一票否决先扫**：标题党词（震惊/速看）、政治敏感、人身攻击、虚假承诺——先扫一遍再评分。
5. **不内置 A/B 测试机制**：本 Agent 输出 Top 3 标题供人工选 1 个最终使用。不做平台 A/B 配置。

### 0.3 与大纲生成 Agent 的核心差异

| 维度 | 大纲生成 | 标题生成 |
|------|---------|---------|
| 候选数量 | 3 个 | **10-15 个** |
| 强制差异化 | 3 种不同开头钩子 | **6-8 种不同套路** |
| 自检维度 | 6 维度（含传播标签完整度） | 6 维度（含**与大纲一致性**） |
| 单次成本预算 | ~0.3 USD | **~0.15 USD**（标题成本低） |
| 失败处理 | 重生 1 次后丢回人工 | 重生 1 次后丢回人工 |
| 最终输出 | 1 个最终大纲 | **3 个推荐标题（人工选 1）** |

## 1. 整体架构

### 1.1 4 Agent 协作图

```
                输入：选题 + 已通过自检的最终大纲
                              │
                              ▼
┌──────────────────────────────────────────────────────────┐
│  Agent A  标题创作员（百万粉 AI 博主视角）                  │
│  ✓ 必须产出 10-15 个标题候选                                │
│  ✓ 必须覆盖至少 6 种不同套路（套路分布强制约束）             │
│  ✓ 每个候选必须标注：套路名、修饰元素、字数、简短说明        │
└──────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────┐
│  Agent B  标题评审员（公众号运营专家视角）                  │
│  ✓ Step 1：一票否决扫描（淘汰命中禁用词的候选）             │
│  ✓ Step 2：6 维度评分（每个候选打分）                       │
│  ✓ Step 3：与大纲一致性交叉验证                             │
│  ✓ Step 4：筛选 Top 5 候选                                  │
└──────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────┐
│  Agent C  读者点击预测员（目标读者代入）                    │
│  ✓ 模拟订阅号信息流场景（标题被一堆标题包围）                │
│  ✓ 对 Top 5 逐个预测点击意愿（0-10）                        │
│  ✓ 输出"会点的原因 / 不会点的原因"                          │
└──────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────┐
│  Agent D  最终判定员                                       │
│  ✓ 综合 Agent B 评分 + Agent C 点击预测                    │
│  ✓ 输出 Top 3 推荐标题（带综合评分 + 推荐理由）              │
│  ✓ 自检：Top 3 综合分 ≥ 7 ？                                │
│                                                            │
│  ≥ 7 → ✅ 通过，输出最终推荐                               │
│  < 7 → 把扣分理由喂给 Agent A 重生（最多 1 次）             │
│  1 次仍不过 → ❌ 丢回人工，标记"难以成标题"                 │
└──────────────────────────────────────────────────────────┘
                              │
                              ▼
                  输出：3 个推荐标题（人工最终选 1）
```

## 2. Agent A - 标题创作员

### 2.1 角色定位
**百万粉 AI 公众号博主**，深谙公众号读者打开决策机制，能从一条选题+大纲里生成多角度、多套路的标题候选。

### 2.2 输入
```yaml
选题:
  标题（草标题，来自选题阶段）: ...
  方向: ...
  套路（选题套路）: ...
  价值承诺: ...
最终大纲:
  各节小标题: ...
  关键信息点: ...
  传播标签分布: ...
```

### 2.3 核心工作规则

**规则 1 - 必须产出 10-15 个候选**
- 10 是下限（保证套路多样性）
- 15 是上限（再多会摊薄质量、增加 token 成本）
- 默认 12 个最舒服

**规则 2 - 强制套路分布**
- 必须覆盖**至少 6 种**不同套路（来自标题套路库的 12 个套路）
- 单一套路下的候选数量不允许超过 **3 个**
- 必须**优先使用**"套路与方向匹配表"中该方向的优先套路（占比 ≥ 70%）
- 允许 1-2 个"慎用套路"尝试（探索空间）

**规则 3 - 每个候选必须标注**
```yaml
- 标题: ...
  套路: ...（来自套路库的 12 种之一）
  修饰元素: [数字, 身份代入, ...]（来自套路库第 3 章的 6 类元素）
  字数: ...
  简短说明: 为什么这样写
```

**规则 4 - 字数硬约束**
- 14-20 字得满分
- 10-13 字或 21-22 字允许但扣分
- < 10 字或 > 22 字禁止输出

**规则 5 - 不允许出现的内容**
- "震惊!""速看""所有人都不知道"等典型标题党词
- 政治/监管敏感表达
- 攻击具体个人/团队
- "包过""保证 100%"等虚假承诺
- 完全文字雷同（仅改一个字算同一个候选）

### 2.4 提示词框架

```
你是一位百万粉的 AI 公众号博主，擅长根据选题和大纲生成多套路的标题候选。

【输入】
选题: {selection}
最终大纲: {outline}

【参考资产】
- 标题套路库：12 种套路 + 6 类修饰元素 + 套路与方向匹配表
  （提示词附带套路库要点）

【你的任务】
产出 10-15 个标题候选，每个候选必须标注使用的套路和修饰元素。

【硬约束】
1. 候选数量 10-15 个
2. 必须覆盖至少 6 种不同套路
3. 单一套路不超过 3 个候选
4. 优先使用该方向的优先套路（占比 ≥ 70%）
5. 每个标题字数 14-20 为最佳（10-13 或 21-22 允许但减分）
6. 不允许标题党词、敏感内容、虚假承诺
7. 文字必须真实差异，不允许"换一个字"的伪候选

【自检清单】
□ 候选数量是否在 10-15？
□ 是否覆盖至少 6 种套路？
□ 单套路是否未超过 3 个？
□ 优先套路占比是否 ≥ 70%？
□ 字数是否都在 14-22？
□ 是否避免了一票否决词？
□ 每个候选是否真实差异？

【输出格式】
（参考第 6 章定义的标准格式）
```

### 2.5 重生模式（第二次调用时）

第一次未通过自检时，Agent A 重新调用，附加扣分理由：

```
【第二次调用，前次失败原因】
{deduction_reasons}

【请针对性改进】
针对上述问题重新生成 10-15 个候选，但仍要保持套路覆盖度。
```

## 3. Agent B - 标题评审员

### 3.1 角色定位
**公众号运营专家**，能从一堆候选中识别"哪些是真正有打开潜力的"。

### 3.2 输入
- Agent A 输出的 10-15 个候选
- 原始选题 + 最终大纲

### 3.3 核心工作（4 步）

**Step 1 - 一票否决扫描**
逐个候选扫描，命中以下任一条件直接淘汰：
- 含标题党词（震惊/速看/所有人都不知道等）
- 涉及政治/监管敏感
- 人身攻击具体个人/团队
- 虚假承诺
- 字数 < 10 或 > 22

**Step 2 - 6 维度评分**
对未被淘汰的候选，按下述维度逐项打分（0-10）：

| 维度 | 权重 |
|------|------|
| 三个一眼达标度 | 25% |
| 情绪触发力度 | 20% |
| 具体性 | 15% |
| 长度合规 | 10% |
| 套路成熟度 | 15% |
| 与大纲一致性 | 15% |

详细锚点见第 5 章。

**Step 3 - 与大纲一致性交叉验证**
检查标题承诺是否能被大纲兑现：
- 标题说"3 个用法"，大纲里是否真有 3 个用法节？
- 标题说"我做了 X"，大纲里是否真讲了 X 的过程？
- 标题暗示反差，大纲里是否真有反差？

不一致的直接降为 0 分（在"与大纲一致性"维度上）。

**Step 4 - 筛选 Top 5**
按加权总分降序，输出 Top 5 候选（含每条的 6 维度分数）。

### 3.4 提示词框架

```
你是一位运营过百万粉公众号的资深运营专家。

【输入：候选标题】
{candidates}

【输入：选题 + 大纲】
{selection}
{outline}

【你的任务】
按 4 步流程评审候选标题，输出 Top 5：

【Step 1 - 一票否决】
扫描候选，命中以下任一条件直接淘汰：
- 含"震惊""速看""所有人都不知道"等标题党词
- 政治/监管敏感
- 人身攻击
- 虚假承诺（包过/100% 保证等）
- 字数 < 10 或 > 22

【Step 2 - 6 维度评分】
对存活候选打分：
- 三个一眼达标度 (25%)
- 情绪触发力度 (20%)
- 具体性 (15%)
- 长度合规 (10%)
- 套路成熟度 (15%)
- 与大纲一致性 (15%)

【Step 3 - 与大纲一致性交叉验证】
标题承诺必须能被大纲兑现。不一致的"与大纲一致性"维度直接 0 分。

【Step 4 - 筛选 Top 5】
按加权总分降序输出 Top 5，每条附 6 维度分数和评分说明。

【输出格式】
（参考第 6 章定义的标准格式）
```

## 4. Agent C - 读者点击预测员

### 4.1 角色定位
**目标读者本人**——28 岁互联网产品经理，男性，订阅了"路人甲TM"，每天打开微信会刷一遍订阅号信息流。

### 4.2 输入
- Agent B 筛选的 Top 5 标题
- 选题 + 大纲（用于上下文，但 Agent C 假装自己只看标题）

### 4.3 核心工作

**模拟场景**：把 Top 5 标题放在订阅号信息流里，周围是其他 5-10 篇日常文章（科技/职场/财经类）。读者在 30 秒内决定"打开哪些"。

**对每个标题预测**：
- 点击意愿（0-10）：看到这个标题，我打开的可能性
- 点击原因（如果会点）：什么吸引了我
- 不点原因（如果不会点）：什么让我划走
- 改进建议：如何让这个标题点击率更高

### 4.4 输出格式

```yaml
点击预测:
  - 标题: ...
    点击意愿: 8
    会点原因: "反差感强 + 有具体数字 + 跟我的工作场景相关"
    改进建议: "把'5 个订阅'改成具体的服务名（如'5 个 SaaS'），更具体"

  - 标题: ...
    点击意愿: 4
    不点原因: "感觉是另一篇 Claude 教程，已经看过类似的"
    改进建议: "需要叠加'反共识'或'数字冲击'修饰元素"
  ...
```

### 4.5 提示词框架

```
你的身份：28 岁互联网产品经理，男性，工作 5 年。
日常使用 Claude / ChatGPT / Cursor 等 AI 工具。
订阅了"路人甲TM"和其他 10+ 科技/职场公众号。
每天早上和晚上会刷一遍订阅号信息流，平均花 30 秒决定打开哪些。

【场景】
你现在正在刷订阅号信息流。"路人甲TM"今天有一篇新文章，你看到了下面这 5 个候选标题之一（每次只能看到 1 个，周围是其他 10 篇日常公众号文章）。

【候选标题】
{top_5_titles}

【你的任务】
逐个预测每个标题在你身上的点击意愿（0-10），并说明会点/不会点的原因。

【真实即可】
请用你的真实反应回答，不要为了"政治正确"给所有候选打高分。
如果某个标题让你瞬间想划走，请直说"看到就划"。
```

## 5. Agent D - 最终判定员 + 6 维度评分锚点

### 5.1 角色定位
综合决策者：把 Agent B 的评分和 Agent C 的点击预测综合，输出 Top 3 推荐。

### 5.2 评分锚点（Agent B 使用）

#### 维度 1 - 三个一眼达标度（权重 25%）

| 分数 | 标准 |
|------|------|
| 9-10 | 1 秒内三个一眼全到位：讲什么、对我什么用、和我什么关系 |
| 7-8 | 三个一眼达成 2 个，第 3 个隐含但需要 2 秒判断 |
| 4-6 | 只达成 1 个一眼（如知道讲什么，但不知道对我有没有用）|
| 1-3 | 三个一眼都模糊，需要点开才知道 |

#### 维度 2 - 情绪触发力度（权重 20%）

| 分数 | 标准 |
|------|------|
| 9-10 | 强烈触发 2 个以上情绪（如焦虑+反差） |
| 7-8 | 明确触发 1 个情绪，力度足 |
| 4-6 | 弱触发 1 个情绪，需要内容补强 |
| 1-3 | 无明显情绪触发，纯陈述句 |

#### 维度 3 - 具体性（权重 15%）

| 分数 | 标准 |
|------|------|
| 9-10 | 含 3 个及以上具体元素（数字/工具名/身份/场景） |
| 7-8 | 含 2 个具体元素 |
| 4-6 | 含 1 个具体元素 |
| 1-3 | 无具体元素，纯抽象表达 |

#### 维度 4 - 长度合规（权重 10%）

| 分数 | 标准 |
|------|------|
| 10 | 14-20 字 |
| 8 | 10-13 字（偏短但可用） |
| 6 | 21-22 字（临界，部分手机截断风险） |
| 1 | < 10 字 或 > 22 字 |

#### 维度 5 - 套路成熟度（权重 15%）

| 分数 | 标准 |
|------|------|
| 9-10 | 命中套路库中已被实战验证的高爆款模式，且使用准确 |
| 7-8 | 命中套路库套路，使用基本到位 |
| 4-6 | 命中套路但生硬（如硬套数字、强行反差） |
| 1-3 | 不属于已知套路，且没有创新感 |

#### 维度 6 - 与大纲一致性（权重 15%）

| 分数 | 标准 |
|------|------|
| 9-10 | 标题承诺被大纲完美兑现，且大纲里有标题的核心元素 |
| 7-8 | 标题承诺被大纲兑现，但大纲对应弱化 |
| 4-6 | 标题承诺被大纲部分兑现 |
| 1-3 | 标题与大纲明显脱节（典型"标题党"） |
| 0 | 标题承诺无法被大纲兑现（直接 0 分） |

### 5.3 加权公式

```
B 评分总分 = 0.25×一眼 + 0.20×情绪 + 0.15×具体性 + 0.10×长度 + 0.15×套路 + 0.15×大纲一致
```

### 5.4 Agent D 的综合公式

```
综合评分 = 0.6 × B评分总分 + 0.4 × C点击预测
```

> 💡 给 B 评分更高权重（60%）是因为 B 的评分基于明确标准，鲁棒性更强；C 的点击预测带个人色彩，作为补充。

### 5.5 通过门槛与重生

- **Top 3 综合评分 ≥ 7.0**：✅ 通过，输出最终推荐
- **Top 3 中有任一 < 7.0**：触发重生
  - 第 1 次失败 → 把 Agent B 和 Agent C 的扣分理由作为额外输入，重新调用 Agent A
  - 第 2 次失败 → 不再尝试，标记"难以成标题"，丢回人工

## 6. 标题的标准输出格式

### 6.1 Agent A 候选标题格式

```yaml
候选标题列表:
  - 序号: 1
    标题: 我以为 Claude Goal 是噱头，结果它替代了我 5 个订阅
    字数: 20
    套路: 反差反转型
    修饰元素: [第一人称, 反差词, 数字, 工具名]
    简短说明: 用"我以为...结果..."反差结构 + 具体数字 5

  - 序号: 2
    标题: 用了一周 Claude Goal，告诉你哪些钱该花哪些不该
    字数: 19
    套路: 痛点直击型
    修饰元素: [第一人称, 时间, 工具名]
    简短说明: 切"AI 订阅费值不值"的真实纠结
  ...
```

### 6.2 Agent B 评审输出格式

```yaml
一票否决扫描:
  通过: [1, 2, 3, 5, 6, 7, 8, 9, 10, 11, 12]
  淘汰:
    - 序号: 4
      原因: 含"震惊"标题党词

Top 5 候选评分:
  - 序号: 1
    标题: ...
    评分:
```

## 7. 候选标题的强制分布约束（汇总）

### 7.1 数量约束
- 10 ≤ 候选数量 ≤ 15

### 7.2 套路约束
- 必须覆盖至少 6 种不同套路
- 单一套路下不超过 3 个候选
- 优先套路（对应该方向的）占比 ≥ 70%
- 允许 1-2 个"慎用套路"探索

### 7.3 字数约束
- 候选标题字数：14-20 为最佳，10-13/21-22 允许但减分
- < 10 或 > 22 禁止输出

### 7.4 修饰元素约束
- 每个候选标注的修饰元素 0-3 个
- 单候选叠加超过 3 个修饰元素的，B 评分会扣分（标题显得啰嗦/浮夸）

### 7.5 差异化约束
- 文字必须真实差异（仅改一个字算同一候选，不计数）
- 多候选若使用同一套路，必须用不同的修饰元素组合区分

## 8. 完整工作流与失败处理

### 8.1 主流程时序

```
T+0    输入：选题 + 最终大纲
T+1    Agent A 第 1 次调用 → 10-15 个候选
T+2    Agent B 评审（一票否决 → 评分 → 大纲一致性 → Top 5）
T+3    Agent C 模拟读者，对 Top 5 预测点击意愿
T+4    Agent D 综合评分 → Top 3 + 自检

       ├─ Top 3 综合分 ≥ 7  → ✅ 输出（结束）
       └─ Top 3 综合分 < 7  → 进入重生流程

T+5    Agent A 第 2 次调用（含 B + C 的扣分理由）→ 10-15 新候选
T+6    Agent B + C + D 再跑一遍

       ├─ ≥ 7  → ✅ 输出（结束）
       └─ < 7  → ❌ 标记"难以成标题"，丢回人工
```

### 8.2 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| Agent A 候选 < 10 个 | 重试 1 次；仍不行降级（接受 8 个候选）并记录异常 |
| 套路覆盖 < 6 种 | Agent B 反馈 Agent A 重生（不计入"自检失败"次数） |
| 单套路 > 3 个候选 | Agent B 自动丢弃超出的，按顺序保留前 3 个 |
| Agent B 评分后 Top 5 < 7 分 | 视为自检失败，触发重生 |
| 连续 2 次自检失败 | 丢回人工，记录失败原因 |
| API 调用失败 | 重试最多 3 次 |

### 8.3 输出归档

每条选题的标题生成产物都归档（用于后续训练/复盘）：
- 10-15 个候选（原始）
- 一票否决淘汰列表
- B 评审 Top 5（含分数）
- C 点击预测
- D 最终 Top 3
- 是否重生、重生次数
- 人工最终选定哪个标题

## 10. 维护与迭代

### 10.1 Agent 提示词的优化
- 实战中如果 Agent A 经常给出某类问题（如反共识套路过多），在提示词里加约束
- 每月回顾一次各 Agent 的输出质量

### 10.2 评分锚点的优化
- 实战中如果某个评分维度的锚点不准（人工和 Agent 评分严重不一致），调整锚点描述
- 锚点调整都要记入版本日志

### 10.3 权重的优化
- v1.0 的权重（25/20/15/10/15/15）是基于讨论拍的
- 运行 1-2 个月后，用真实"标题 vs 打开率"数据反推权重
- 比如发现"具体性强的标题打开率显著更高"，可以提高该维度权重

### 10.4 综合公式优化
- 当前是 B 评分 × 60% + C 点击预测 × 40%
- 如果发现 Agent C 的点击预测和真实数据更吻合，可以提高 C 的权重
- 反之降低

### 10.5 反馈回流
每篇文章发布后，回填标题真实数据：
- 实际打开率
- 在订阅号信息流的曝光转化率
- Top 3 中选用的是哪个、为什么选

这些数据每月做回归分析，反向校准评分卡和综合公式。
