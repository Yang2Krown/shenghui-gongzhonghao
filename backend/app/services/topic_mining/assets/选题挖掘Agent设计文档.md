# 选题挖掘 Agent 设计文档 v2.0

> **用途**：定义"针对单条信息进行选题挖掘"的 Agent 架构、工作流、输入输出格式。
> **核心职责**：**对单条信息进行选题衍生 + 评分**。不负责信息预处理（去重/聚类）、不负责全局排序（方向均衡）——这些是上下游脚本层的事。
> **与其他文档的关系**：上游接信息采集与预处理脚本；调用 [选题套路库](./选题套路库.md) 与 [选题评分卡](./选题评分卡.md)；下游输出给"全局排序脚本"，再传到大纲生成 Agent。
> **维护者**：路人甲TM 团队
> **最后更新**：2026-05-20
> **版本说明**：v2.0 大幅重构，从"全局批量处理"改为"单条信息深度挖掘"。

## 0. 设计原则

### 0.1 这个 Agent 解决什么问题
给定**一条经过预处理的信息**（来源、标题、要素、信息类型已知），产出 **3-8 个候选选题 + 每个候选的评分**。

### 0.2 明确的职责边界

**本 Agent 做什么**：
- ✅ 单条信息的选题衍生（一信息多角度）
- ✅ 单条信息的候选选题评分

**本 Agent 不做什么**：
- ❌ 信息去重 / 多源聚类（上游脚本做）
- ❌ 信息类型标注 / 要素提取（上游脚本做）
- ❌ 全局 Top N 排序（下游脚本做）
- ❌ 方向多样性均衡（下游脚本做）
- ❌ 同主题去重（下游脚本做）

> 💡 为什么这样划分？因为去重、排序、均衡这些是**规则性工作**，用脚本做更稳定、更便宜、更可解释。Agent 的核心价值在**"创造性衍生 + 多维度判断"**——把 LLM 用在 LLM 真正擅长的地方。

### 0.3 三条核心原则

1. **是"创造器"不是"过滤器"**：从一条信息里衍生出读者没想到的角度，是这个 Agent 的核心价值。
2. **多 Persona 内嵌评议**：单一视角容易盲区，所以 Agent A 内部必须做"AI 专家 / 百万粉博主 / 产品经理 / 运营专家"四视角评议。
3. **衍生 + 评分一次跑完**：单条信息处理是"原子操作"，从输入到输出在一次调用链里完成，不需要中间停下等其他信息。

## 1. 整体架构

### 1.1 2 Agent 协作图

```
              输入：1 条预处理后的信息
              （信息类型已标注 / 要素已提取）
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  Agent A  选题衍生员（多 Persona 评议）                     │
│  ✓ 调用《选题套路库》                                       │
│  ✓ Step 1：套路匹配（按信息类型匹配对应章节套路）            │
│  ✓ Step 2：8 维度组合创新（生成非常规角度）                  │
│  ✓ Step 3：4 Persona 内嵌评议                              │
│  ✓ 衍生量：资讯型 5-8 / 实操 4-6 / 观点 4-6 / 教程 3-5     │
└──────────────────────────────────────────────────────────┘
                          │
                  候选选题（含 Persona 评议）
                          ▼
┌──────────────────────────────────────────────────────────┐
│  Agent B  选题评分员                                       │
│  ✓ 调用《选题评分卡》                                       │
│  ✓ Step 1：一票否决扫描                                     │
│  ✓ Step 2：6 维度评分（含子项加权）                         │
│  ✓ Step 3：加权总分 + 入选判定（入选 / 备选 / 淘汰）         │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
              输出：候选选题清单
              （每条含套路、评分、判定、衍生路径）
```

### 1.2 数据流

```
1 条预处理信息
   ↓
3-8 个候选选题（Agent A 衍生）
   ↓
每条带 6 维度评分 + 入选判定（Agent B 评分）
   ↓
输出供下游脚本消费
```

### 1.3 性能预算（参考）

| Agent | 模型 | 单次预估 token | 备注 |
|-------|------|--------------|------|
| Agent A | Claude/GPT 主模型 | ~3500 (in) + ~3500 (out) | 多 Persona，输出 3-8 候选 |
| Agent B | 同上 | ~3000 (in) + ~2500 (out) | 每候选评分 |
| **总计** | | ~6500 (in) + ~6000 (out) | 单条信息处理 |

预算约 **0.15-0.25 USD/条信息**（按 Claude Sonnet 价格）。

> 💡 实际成本随候选数量浮动。资讯型信息因为衍生量大（5-8 个），成本会偏高。

## 2. Agent A - 选题衍生员

### 2.1 角色定位
**百万粉博主 + AI 专家 + 产品经理 + 运营专家** 的合体——从一条信息里看出多个选题角度。这是整个 pipeline 的"创造力"所在。

### 2.2 输入
- 1 条预处理后的信息（见 1.4 节格式）
- 《选题套路库》（作为提示词附带的参考资产）

### 2.3 核心工作（3 步）

**Step 1 - 套路匹配（基于套路库）**
根据信息类型，找到套路库对应章节，**逐个套路套用**：
- 资讯型 → 1.1 章 7 个套路
- 实操案例型 → 1.2 章 6 个套路
- 观点分享型 → 1.3 章 6 个套路
- 教程型 → 1.4 章 6 个套路

每个套路生成 1 个候选选题（标题草稿 + 价值承诺 + 切入说明）。

**Step 2 - 8 维度组合创新（基于套路库第 2 章）**
为了避免选题千篇一律，从 8 大切入维度（价值/身份/态度/时间/比较/故事/反差/结构）中抽取 2-3 个组合，生成"非常规"候选。每条信息至少 2-3 个组合型候选。

**Step 3 - 4 Persona 内嵌评议**
对所有衍生候选，由 4 个 Persona 分别打分（0-10）+ 简短理由：

| Persona | 关注点 |
|---------|--------|
| **AI 专家** | 技术价值、信息准确性、有没有理解错 |
| **百万粉博主** | 标题感、爆款基因、传播路径 |
| **产品经理** | 解决了什么痛点、价值是否真实 |
| **运营专家** | 时机、节奏、是否商务敏感 |

**特别标注**：4 Persona 分歧度高（max-min ≥ 4）的候选标 ⚠️ —— 这些是"最有潜力或最危险"的选题，下游 review 时要重点看。

### 2.4 衍生量约束

| 信息类型 | 衍生量 | 说明 |
|---------|--------|------|
| 资讯型 | 5-8 个 | 套路多、角度丰富 |
| 实操案例型 | 4-6 个 | 重点深挖复刻/拆解 |
| 观点分享型 | 4-6 个 | 正反两面都要切 |
| 教程型 | 3-5 个 | 变体相对少 |

### 2.5 输出格式

```yaml
候选选题列表:
  - 候选ID: T-001
    源信息ID: cluster-001
    标题: Claude Goal 真正可怕的，是这 3 个能力的组合
    方向: 观点型
    套路: 1.1.1 深度解读型
    维度组合: [态度=吹爆, 结构=列表]
    价值承诺: 让读者获得"看懂内行门道"的优越感
    切入说明: 剥开发布通稿，剖析 3 个关键能力
    Persona 评议:
      AI 专家: 8 - 技术解读靠谱
      百万粉博主: 9 - "真正可怕"有钩感
      产品经理: 7 - 切中"想看懂"的需求
      运营专家: 8 - 时机正好（发布后 3 天内）
    Persona 分歧度: 低 (max-min = 2)

  - 候选ID: T-002
    源信息ID: cluster-001
    标题: 所有人都在吹 Claude Goal，但我说它就是个噱头
    方向: 观点型
    套路: 1.1.7 反共识型
    维度组合: [态度=反共识, 故事=第一人称]
    价值承诺: 让读者获得"独立思考者"的认同感
    切入说明: 用反共识立场吸引"独立思考"读者
    Persona 评议:
      AI 专家: 6 - 反共识需要论据支撑
      百万粉博主: 10 - 反共识标题打开率高
      产品经理: 7 - 切中"独立思考"心理
      运营专家: 5 - 商务敏感，需谨慎
    Persona 分歧度: 高 (max-min = 5) ⚠️
  ...
```

## 3. Agent B - 选题评分员

### 3.1 角色定位
独立评分员，按《选题评分卡》对每个候选打分，输出哪些选题应该入选。

### 3.2 输入
- Agent A 输出的候选选题列表
- 《选题评分卡》（作为提示词附带的参考资产）

### 3.3 核心工作（3 步）

**Step 1 - 一票否决扫描**
逐个候选扫描，命中任一条件直接淘汰：
- 受众适配度 < 3
- 痛点直击度 < 2 且价值密度 < 2
- 政治/监管敏感
- 灰产/敏感行业
- 人身攻击
- 纯标题党
- 严厉唱衰具体国内厂商 → 标记"商务敏感，人工复核"（不直接淘汰）

**Step 2 - 6 维度评分**
对存活候选打分（参考评分卡锚点）：

| 维度 | 权重 |
|------|------|
| 痛点直击度 | 20% |
| 价值密度 | 20% |
| 传播触发器 | 15% |
| 差异化 | 15% |
| 新鲜度 | 10% |
| 受众适配度 | 20% |

**Step 3 - 加权总分 + 入选判定**
- 总分 ≥ 7.0：✅ 入选
- 5.0 - 7.0：⏳ 备选
- < 5.0：❌ 淘汰

### 3.4 输出格式

```yaml
候选选题清单（评分后）:
  - 候选ID: T-001
    源信息ID: cluster-001
    标题: Claude Goal 真正可怕的，是这 3 个能力的组合
    方向: 观点型
    套路: 1.1.1 深度解读型
    维度组合: [态度=吹爆, 结构=列表]
    价值承诺: 让读者获得"看懂内行门道"的优越感

    一票否决: 通过
    评分:
      痛点直击度:
        分数: 7
        依据: 切中"想看懂内行门道"的真实诉求
      价值密度:
        分数: 8
        依据: 能给出 3 个具体能力分析
      传播触发器:
        分数: 8
        依据: "真正可怕"+数字 3，触发好奇+焦虑
      差异化:
        分数: 7
        依据: 同类深度解读较少
      新鲜度:
        分数: 9
        依据: 发布 3 天内
      受众适配度:
        分数: 9
        依据: 工具主流 + 受众完全匹配

    加权总分: 7.95
    判定: ✅ 入选

    Persona 分歧度: 低
    衍生路径:
      源信息: cluster-001
      套路: 1.1.1 深度解读型
      维度组合: [态度=吹爆, 结构=列表]

  - 候选ID: T-002
    ...
    一票否决: 通过 (但商务敏感，标记人工复核)
    加权总分: 7.05
    判定: ⏳ 备选 (商务敏感 + 评分边缘)
  ...

清单统计:
  总候选数: 6
  入选: 3
  备选: 2
  淘汰: 1
  含商务敏感标注: 1
  含高 Persona 分歧度: 1
```

## 4. 完整工作流

### 4.1 主流程时序

```
T+0   输入：1 条预处理后的信息
T+1   Agent A 衍生 → 3-8 个候选选题（含 Persona 评议）
T+2   Agent B 评分 → 候选选题清单（含判定）
T+3   输出
```

### 4.2 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| Agent A 衍生数量低于该类型下限 | 重试 1 次；仍不行降级输出已生成的候选 |
| Agent A 输出格式不符合 schema | 重试 1 次，仍失败标记任务失败 |
| Agent B 评分缺失某个维度 | 视为评分异常，对该候选重新打分 |
| 全部候选都被一票否决淘汰 | 不算异常，输出空清单（说明该信息不适合做选题） |
| API 调用失败 | 重试最多 3 次 |

### 4.3 归档

每次单条信息处理完，归档：
- 输入的预处理信息
- Agent A 输出的所有候选（含 Persona 评议）
- Agent B 评分明细
- 一票否决淘汰记录
- 最终判定结果

## 6. 维护与迭代

### 6.1 Agent 提示词的优化
- 实战中如果 Agent A 衍生过于集中（如总是用"反共识"套路），加约束
- 实战中如果某个 Persona 评议偏差大（如运营专家总是低分），调整其评议提示词
- 每月回顾各 Agent 输出质量

### 6.2 评分锚点的优化
- 实战中如果某个评分维度的锚点不准（人工和 Agent 评分严重不一致），更新评分卡
- 锚点调整记入版本日志

### 6.3 衍生量约束的优化
- 当前各类型的衍生量是基于经验拍的
- 运行 1-2 个月后，根据"哪类信息真的能撑住更多角度"调整

### 6.4 多 Persona 权重的优化
- v2.0 中 4 Persona 平权评议（仅用于标注分歧度）
- 未来可以加权——比如发现"百万粉博主"的评议预测准确率最高，可以让其评议结果对评分有更大影响

### 6.5 反馈回流
每周回顾：
- 入选的候选实际表现如何（发布后阅读/点赞）
- 备选的候选如果被人工 override 入选，表现如何
- 一票否决的候选有没有被误杀
- 高 Persona 分歧度的候选最终结果如何

这套数据反向校准评分卡、套路库、Persona 权重。
