# 正文生成模块设计文档

> **模块**：`backend/app/services/content_generation/`
> **上游**：选题挖掘 → 大纲生成 → 标题生成 → **正文生成**
> **下游**：人工 review → 公众号发布
> **设计参考**：《正文生成 Agent 设计文档 v1.1》+ 《去 AI 味检查清单 v1.0》
> **日期**：2026-05-22

---

## 1. 架构概览

正文生成采用 **4 Agent 串行协作** 模式，与选题挖掘流程完全独立，不干扰现有选题 pipeline。

```
输入：选题 + 大纲 + 标题 + 风格参数
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│  Agent A  正文创作员（整篇生成）                           │
│  ✓ 一次性生成 2500-3000 字                                │
│  ✓ 严格按大纲展开（每节字数配额）                          │
│  ✓ 兑现标题承诺                                           │
│  ✓ 风格锚点机制（先定义本文语气，再分节写）                │
│  ✓ 在原文里自然产生 1-2 个"原生金句"作为种子              │
└─────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│  Agent B  金句催化员                                      │
│  ✓ 识别正文中的"准金句"位置                                │
│  ✓ 催化生成 3-5 个金句（含位置标注、类型标签、不可改标签） │
│  ✓ 类型分布：开头钩子 + 中段共鸣 + 结尾升华               │
│  ✓ 保护原文骨干（不重写正文，只增强/补足金句）             │
└─────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│  Agent C  去 AI 味改写员                                   │
│  ✓ 调用《去 AI 味检查清单》                                │
│  ✓ 全文扫描，按 6 大类 AI 味识别                           │
│  ✓ 按优先级（🚫⚠️⚪🔵）改写                                │
│  ✓ 跳过金句标注的部分（不允许改）                          │
│  ✓ 改写后字数变化 ≤ ±10%                                  │
└─────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│  Agent D  整合 + 自检诊断员                                │
│  ✓ 将金句精准嵌入对应位置                                  │
│  ✓ 8 维度自检评分（仅评分 + 给建议，不触发重生）           │
│  ✓ 输出结构化诊断报告（含改进建议优先级）                  │
└─────────────────────────────────────────────────────────┘
         │
         ▼
输出：
- 最终正文（2500-3000 字）
- 金句清单（含位置标注 + 不可改标签）
- 自检诊断报告（8 维度评分 + 具体改进建议）
- 去 AI 味改写对照表（供人工 review 参考）
```

## 3. 与现有模块的关系

### 3.1 不影响选题流程

正文生成模块与选题挖掘模块**完全独立**：

| 维度 | 选题挖掘 | 正文生成 |
|------|---------|---------|
| 服务目录 | `services/topic_mining/` | `services/content_generation/` |
| Celery 任务 | `topic_mining_tasks.py` | `content_generation_tasks.py` |
| API 端点 | `/api/v1/topic-candidates/` | `/api/v1/content-generation/` |
| 数据模型 | `TopicCandidate`, `InfoCluster` | `ContentCreation`, `Outline` |
| 触发时机 | 信息簇预处理后 | 选题+大纲确定后 |

### 3.2 数据流

```
InfoCluster → [选题挖掘] → TopicCandidate
                                │
                                ▼
                           [大纲生成] → Outline
                                │
                                ▼
                           [标题生成] → 最终标题
                                │
                                ▼
TopicCandidate + Outline + 标题 → [正文生成] → ContentCreation
```

### 3.3 共享依赖

- **LLM 客户端**：`app.services.llm.get_llm_client()`
- **数据库会话**：`app.db.session.SessionLocal`
- **Celery 应用**：`app.core.celery_app.celery_app`

## 4. API 接口

### 4.1 异步生成（推荐）

```
POST /api/v1/content-generation/generate
```

**请求体**：
```json
{
  "candidate_id": 123,
  "outline_id": 456,
  "style_params": {
    "tone": "第一人称、带点自嘲、信息密集",
    "banned_words": ["赋能", "颠覆"],
    "preferred_words": ["干货", "实操"]
  }
}
```

**响应**：
```json
{
  "code": 200,
  "message": "正文生成任务已提交",
  "data": {
    "task_id": "abc-123-def",
    "status": "PENDING"
  }
}
```

### 4.2 同步生成（调试用）

```
POST /api/v1/content-generation/generate/sync
```

直接传入选题+大纲数据，不查库，适合预览和调试。耗时约 30-60 秒。

### 4.3 查询任务状态

```
GET /api/v1/content-generation/task/{task_id}
```

## 5. Agent 详细设计

### 5.1 Agent A — 正文创作员

**职责**：一次性生成 2500-3000 字正文骨干

**核心规则**：
1. 整篇生成（一次产出，不分节生成）
2. 风格锚点机制（先定义本文语气，再分节写）
3. 严格按大纲展开（每节小标题不允许重写）
4. 兑现标题承诺
5. 字数硬性配额（总 2500-3000，单节 400-600）
6. 原生金句种子（开头节末尾、结尾节开头各埋 1 个）
7. 提前避免最严重的 AI 味

**输出**：
```yaml
style_anchor: "本文语气：第一人称、带点自嘲、信息密度高"
full_text: "完整正文..."
total_word_count: 2680
section_count: 5
sections: [...]
gold_seeds: [...]
```

### 5.2 Agent B — 金句催化员

**职责**：催化 3-5 个金句，标注位置 + 不可改标签

**金句类型分布**：
- 开头钩子金句：1 个（第 1 节内）
- 中段共鸣金句：1-2 个（第 2-3 节）
- 结尾升华金句：1 个（最后一节）
- 反差/反共识金句：0-1 个（任意位置）

**金句质量要求**：
- 字数 15-50
- 含反差/抽象提炼/共鸣表达/强观点
- 脱离上下文也能独立成立
- 不是流于表面的鸡汤

### 5.3 Agent C — 去 AI 味改写员

**职责**：按《去 AI 味检查清单》净化全文

**6 大类 AI 味**：
1. 结构性（连接词、套话、对称）
2. 词汇性（学院体、程度副词、营销话术）
3. 句式性（长句、对仗、被动）
4. 情感性（情感缺失、立场缺失、故事感缺失）
5. 思维性（概括、平均、完美）
6. 排版性（双引号、列表、加粗、emoji）

**优先级体系**：
- 🚫 一票否决：出现即必须改
- ⚠️ 高优先级：每篇 ≤ 1 次
- ⚪ 中优先级：每篇 ≤ 3 次
- 🔵 低优先级：酌情改

**保护原则**：
- 金句标注段落：✋ 跳过
- 含具体数据/人名/工具名：✋ 跳过
- 直接引用：✋ 跳过

### 5.4 Agent D — 整合 + 自检诊断员

**职责**：金句嵌入 + 8 维度评分 + 诊断报告

**8 维度评分**：

| 维度 | 权重 | 评分依据 |
|------|------|---------|
| 标题承诺兑现度 | 20% | 标题说的事正文里都给到了吗 |
| 大纲结构对应度 | 15% | 每节是否按大纲展开 |
| 字数合规 | 10% | 总字数 + 单节字数是否合规 |
| 风格统一性 | 15% | 全文语气是否一致 |
| 去 AI 味彻底度 | 15% | 是否还有残留的 AI 味 |
| 金句完整度 | 10% | 3-5 个金句是否到位 |
| 开头质量 | 10% | 前 200 字是否抓人 |
| 结尾升华度 | 5% | 是否避免烂尾 |

**处理路径建议**：
- 接受发布（总分 ≥ 8）
- 局部手改（总分 6-8）
- 整篇重写（总分 < 6）

## 6. 成本预算

| Agent | 模型 | 单次预估 token | 备注 |
|-------|------|--------------|------|
| Agent A | DeepSeek/Claude | ~3000 (in) + ~4000 (out) | 长输出，主要成本 |
| Agent B | 同上 | ~4500 (in) + ~800 (out) | 只生成金句，输出小 |
| Agent C | 同上 | ~5500 (in) + ~5500 (out) | 去 AI 味需要重写全文 |
| Agent D | 同上 | ~6000 (in) + ~5000 (out) | 整合输出 + 评分诊断 |
| **总计** | | ~19000 (in) + ~15300 (out) | 单次完整流程 |

预算约 **1.3-1.5 USD/篇**（按 Claude Sonnet 价格）。

## 7. 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| Agent A 字数 < 2000 | 报告里标"字数严重不足"，建议丢回 |
| Agent A 字数 > 3500 | 报告里标"字数超额"，建议截取或精简 |
| Agent A 漏掉大纲某节 | 报告里标"大纲结构对应度"为 4 分 |
| Agent B 金句数 < 3 | 报告里标"金句完整度"低，建议人工补 |
| Agent C 字数变化超 ±10% | 视为改写异常，回退到改写前版本 |
| Agent C 改写破坏了金句 | 视为重大异常，直接回退 |
| Agent D 自检总分 < 5 | 报告里标"建议整篇重写" |
| API 调用失败 | 重试 1 次，仍失败标记任务失败 |
